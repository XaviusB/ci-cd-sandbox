== Créer une application Flask simple

=== Créer les fichiers de base

* [ ] Créer un fichier nommé `main.py` avec le contenu suivant :

```python
import os
from flask import Flask, jsonify
from threading import Thread

app = Flask(__name__)

@app.route('/')
def hello_world():
    print("The root endpoint was accessed!")
    return "Hello, World!"

@app.route('/healthz')
def health_check():
    print("The health_check endpoint was accessed!")
    response = jsonify(status='OK')
    response.headers['X-Custom-Header'] = 'Awesome'
    return response, 200

def run_flask():
    app.run(host='0.0.0.0', port=int(os.environ.get('PORT', 8080)))

if __name__ == '__main__':
    flask_thread = Thread(target=run_flask)
    flask_thread.start()
```

* [ ] Créer un fichier requirements.txt avec le contenu suivant :

```
blinker==1.9.0
click==8.1.8
Flask==3.1.0
itsdangerous==2.2.0
Jinja2==3.1.5
MarkupSafe==3.0.2
Werkzeug==3.1.3
```

> Voilà, nous avons une application Flask simple qui expose deux endpoints : `/` et `/healthz`. Le premier retourne un message "Hello, World!" et le second retourne un statut de santé avec un en-tête personnalisé.

== Création d'une image Docker

* [ ] Créer un fichier `Dockerfile` à la racine de votre projet

```Dockerfile
FROM python:3.10-slim

WORKDIR /app
RUN apt update && apt -y install curl
COPY requirements.txt /app/requirements.txt
RUN  pip install -r requirements.txt
COPY main.py /app/main.py

ENTRYPOINT [ "python", "main.py" ]
```

Puis build l'image Docker localement :

```sh
docker build -t <nom_de_l'étudiant>-api:latest .
docker run -p 8080:8080 --rm --name test <nom_de_l'étudiant>-api:latest
```

Vous pouvez vous connecter en bash à votre conteneur

```sh
docker exec -it test bash
```

Une fois l'image prête, vérifier sa taille

```sh
docker image ls
```

> FinOps

* [ ] Trouver des moyens de réduire la taille de votre image Docker pour optimiser les coûts de stockage et de déploiement. L'image doit faire moins de 100Mo. Curl doit demeurer sur l'image pour les tests de santé.

* [ ] Configurer le cache de pip dans votre pipeline CI/CD pour accélérer l'installation des dépendances en réutilisant les paquets déjà téléchargés.

> Secops

* [ ] Trouver un moyen pour le code puisse être executé mais pas être modifié à l'intéreur de l'image

=== Build d'image Docker

* [ ] Dans gitlab.com créer un token `Settings > Repository > Deploy Tokens` avec les permissions `read_registry` et `write_registry` pour pouvoir pousser votre image Docker vers le registre de conteneurs de GitLab.

> Ne perdez pas ce token et ne pas le _commit_ dans votre code.

* [ ] Ajouter les variables d'environnement `CI_REGISTRY_USER` et `CI_REGISTRY_PASSWORD` dans les paramètres de votre projet GitLab `Settings > CI / CD > Variables` pour stocker les informations d'identification nécessaires à l'authentification avec le registre de conteneurs.

* [ ] Identifier le nom de votre image Docker grâce au menu `Deploy > Container Registry` de votre projet GitLab. Le nom de l'image devrait être quelque chose comme `registry.gitlab.com/<votre_nom_d'utilisateur>/<votre_projet>/<nom_de_l'image>`.

* [ ] Tenter de _build_ et de _push_ l'image depuis votre poste

```sh
docker login registry.gitlab.com
docker build -t registry.gitlab.com/<votre_nom_d'utilisateur>/<votre_projet>/<nom_de_l'image>:test .
docker push registry.gitlab.com/<votre_nom_d'utilisateur>/<votre_projet>/<nom_de_l'image>:test
```

* [ ] Ajouter une étape de build d'image Docker dans votre pipeline CI/CD pour construire une image de votre application à partir du `Dockerfile` que vous avez créé.

> Semantic versioning

[bullet]
  * Soit il s'agit d'une pull-request, dans ce cas on tag l'image `pr-<numéro_de_pr>-<sha1_du_commit>`
  * Soit il s'agit d'un tag gitlab, dans ce cas on tag l'image `v<version-du-tag>-<sha1_du_commit>` (le `v` qui précède est une très bonne pratique)

* [ ] Ajouter une étape de push d'image Docker en respectant le _semantic versioning_

* [ ] Vérifier que votre image Docker a été poussée avec succès `Deploy > Container Registry`

=== Scan de vulnérabilités

* [ ] Ajouter une étape de scan de vulnérabilités dans votre pipeline CI/CD après le _build_ et avant le _push_ en utilisant Trivy avant de pousser votre image Docker vers le registre de conteneurs.

| Conseil: vous pouvez utiliser `docker save` pour sauvegarder votre image Docker et ensuite utiliser Trivy pour scanner l'image sauvegardée sans avoir besoin de la pousser vers le registre de conteneurs.


> SecOps

[bullet]
  * Si des vulnérabilités sont détectées, le pipeline doit échouer et afficher les détails des vulnérabilités trouvées.
  * Si aucune vulnérabilité n'est détectée, l'image Docker doit être poussée vers le registre de conteneurs.
  * Il faut corriger les vulnérabilités détectées dans votre image Docker en mettant à jour votre `Dockerfile` ou en utilisant des versions plus récentes des dépendances ou en .trivyignore les vulnérabilités qui ne sont pas exploitables/_fixables_

* [ ] Créer trois _manifests_ Kubernetes dans une nouveau dossier qui sera bientôt un nouveau repo git, en prenant soin de remplacer les valeurs `FIXME` par les valeurs appropriées pour votre projet :

`deployment.yml`: ce manifeste crée une ressource de déploiement Kubernetes qui définit comment votre application Flask doit être déployée sur le cluster Minikube. Assurez-vous de remplacer `FIXME:FIXME` par le nom de votre image Docker dans la section `image` et `FIXME.py` par le nom de votre fichier Python principal (dans ce cas, `main.py`).

La variable d'environnement `PYTHONUNBUFFERED` est définie pour s'assurer que les logs de votre application sont affichés en temps réel dans les journaux Kubernetes, ce qui facilite le débogage et la surveillance de votre application.

```yml
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: awesome-app
  labels:
    app: awesome-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: awesome-app
  template:
    metadata:
      labels:
        app: awesome-app
    spec:
      containers:
        - name: web-app
          image: FIXME:FIXME
          tty: true
          env:
            - name: PYTHONUNBUFFERED
              value: '1'
          imagePullPolicy: Always
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 15
            timeoutSeconds: 1
            failureThreshold: 1
          resources:
            requests:
              cpu: 100m
```

`ingress.yml`: ce manifeste crée une ressource d'ingress qui expose votre application Flask en utilisant le gestionnaire d'ingress NGINX. Assurez-vous que le nom du service dans la section `backend` correspond à celui défini dans votre manifeste de service Kubernetes.

```yml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress
spec:
  ingressClassName: nginx
  rules:
    - host: app-dev.devops.active
      http:
        paths:
          - backend:
              service:
                name: service
                port:
                  number: 8080
            path: /
            pathType: Prefix
```

service.yml: ce manifeste expose votre application Flask en créant un service Kubernetes de type ClusterIP. Assurez-vous que le nom du service correspond à celui utilisé dans votre manifeste d'ingress.

```yaml
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: awesome-app
apiVersion: v1
kind: Service
metadata:
  name: service
spec:
  selector:
    app: awesome-app
  ports:
    - protocol: TCP
      port: 8080
      targetPort: 8080
```

Voici un aperçu de la dependance entre les différentes ressources Kubernetes que nous avons créées :

```
Deployment (awesome-app) --> Service (service) --> Ingress (ingress) --> NGINX Ingress Controller --> LAN/WAN <-- User
```


* [ ] Créer un namespace dédié à votre application pour son environemnt de développement. Un namespace Kubernetes est une manière de partitionner les ressources d'un cluster en plusieurs espaces de noms logiques, ce qui permet de mieux organiser et gérer les ressources pour différentes applications ou environnements.
Les commandes `kubectl -n <namespace>` permettent de spécifier le namespace dans lequel vous souhaitez interagir.
Le namespace par défaut est `default`, mais il est recommandé de créer des namespaces séparés pour chaque application ou environnement pour éviter les conflits de ressources et faciliter la gestion du cluster.

```sh
kubectl create namespace my-app-dev
```

* [ ] Créer une secret Kubernetes pour stocker les informations d'identification de votre registre de conteneurs GitLab

```sh
kubectl create secret docker-registry gitlab-registry-secret \
  --docker-server=registry.gitlab.com \
  --docker-username=$CI_REGISTRY_USER \
  --docker-password=$CI_REGISTRY_PASSWORD \
  -n my-app-dev
```

* [ ] Appliquer les _manifests_ Kubernetes pour déployer votre application sur le cluster Minikube

```sh
kubectl apply -f deployment.yml -f service.yml -f ingress.yml -n my-app-dev
```

* [ ] Vérifier que les ressources Kubernetes ont été créées avec succès


== Passer en multi environnement

Il va falloir remodeliser nos manifestes Kubernetes pour pouvoir les réutiliser dans différents environnements (dev & prod). Nous allons utiliser Kustomize pour cela.

Kustomize est un outil de personnalisation de ressources Kubernetes qui permet de gérer les configurations spécifiques à chaque environnement tout en réutilisant une base commune de manifestes. Il utilise une approche déclarative pour définir les personnalisations, ce qui facilite la maintenance et la gestion des configurations Kubernetes.

* [ ] Créer un dossier `overlays` à la racine de votre projet de déploiement Kubernetes et créer des sous-dossiers pour chaque environnement (dev, prod).

Dans chaque dossier d'environnement, vous pouvez créer des fichiers `kustomization.yaml`

Pour la prod `overlays/prod/kustomization.yaml`:

| Vous remarquerez que nous avons ajouté une section `patches` pour personnaliser l'URL de l'ingress dans l'environnement de production, et une section `images` pour mettre à jour l'image Docker utilisée par le déploiement avec la version appropriée pour chaque environnement.

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: my-app-prod
resources:
  - ../../base/deployment.yml
  - ../../base/service.yml
  - ../../base/ingress.yml
patches:
  - target:
      kind: Ingress
      name: ingress
    patch: |-
      - op: replace
        path: /spec/rules/0/host
        value: app.devops.active
images:
  - name: registry.gitlab.com/<votre_nom_d'utilisateur>/<votre_projet>/<nom_de_l'image>
    newName: registry.gitlab.com/<votre_nom_d'utilisateur>/<votre_projet>/<nom_de_l'image>
    newTag: FIXME
```

Pour la dev `overlays/dev/kustomization.yaml`:

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: my-app-dev
resources:
  - ../../base/deployment.yml
  - ../../base/service.yml
  - ../../base/ingress.yml
images:
  - name: registry.gitlab.com/<votre_nom_d'utilisateur>/<votre_projet>/<nom_de_l'image>
    newName: registry.gitlab.com/<votre_nom_d'utilisateur>/<votre_projet>/<nom_de_l'image>
    newTag: FIXME
```

* [ ] Déplacer vos manifestes Kubernetes dans le dossier `base` et créer des fichiers `kustomization.yaml` dans chaque dossier d'environnement pour définir les personnalisations spécifiques à chaque environnement.

La structure de votre projet devrait ressembler à ceci :
```
/
├── base/
│   ├── deployment.yml
│   ├── service.yml
│   └── ingress.yml
└── overlays/
    ├── dev/
    │   └── kustomization.yaml
    └── prod/
        └── kustomization.yaml
```

* [ ] Commit & Push vos changements vers votre dépôt GitLab.
